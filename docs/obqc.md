# OBQC -- Ontology-Based Query Check

OBQC is a deterministic, rule-based SQL validator that checks every query against the loaded RDF/OWL ontology before it reaches the database. It uses [sqlglot](https://github.com/tobymao/sqlglot) for SQL parsing and the `oba:` namespace annotations in the ontology as its source of truth for tables, columns, types, and relationships.

No LLM calls, no probabilistic reasoning -- OBQC provides hard structural guarantees that complement the LLM's SQL generation.

## When OBQC runs

OBQC activates automatically when an ontology is available in the session:

| Tool | OBQC behavior |
|------|---------------|
| `execute_sql_query` | Runs OBQC first. **Blocks execution** if any ERROR-level issue is found. Attaches warnings to the response if the query passes. |

OBQC is skipped (with an informational message) when:

- No ontology is loaded (`generate_ontology` or `load_my_ontology` has not been called)
- The loaded ontology lacks `oba:` namespace annotations (e.g. a generic OWL ontology not generated by OrionBelt®)

## Severity levels

Each issue has a severity that determines whether the query is blocked or allowed to proceed:

| Severity | Effect in `execute_sql_query` |
|----------|-------------------------------|
| **ERROR** | Query is **blocked** -- not sent to the database. Returns `"success": false` with issue details. |
| **WARNING** | Query **executes normally**. Warnings are attached to the response so the LLM can inform the user or self-correct. |
| **INFO** | Informational note, no effect on execution. |

The overall `is_valid` flag is determined by: **any ERROR-level issue makes the result invalid**.

## Validation rules

### Table existence (ERROR)

Checks that every table referenced in `FROM` and `JOIN` clauses exists in the ontology.

```sql
-- ERROR: Table 'cusotmers' not found in ontology
SELECT * FROM cusotmers;
```

When a table is not found, OBQC suggests up to 10 available table names from the ontology.

### Column existence (ERROR)

Checks that every column reference resolves to an actual column in its table.

```sql
-- ERROR: Column 'emial' not found in table 'customers'
SELECT emial FROM customers;
```

For qualified references (`table.column`), the column is checked against the specific table. For unqualified references, OBQC searches all tables in the query.

### Ambiguous columns (WARNING)

When an unqualified column name exists in multiple tables referenced by the query.

```sql
-- WARNING: Column 'id' is ambiguous - exists in: customers, orders
SELECT id FROM customers JOIN orders ON customers.id = orders.customer_id;
```

OBQC suggests qualifying the column with the table name.

### Join validity

#### Missing join condition (ERROR)

Multiple tables in the query without explicit `JOIN ... ON` conditions (Cartesian product).

```sql
-- ERROR: Multiple tables without explicit JOIN (Cartesian product)
SELECT * FROM customers, orders;
```

#### Non-matching join condition (WARNING)

Join condition does not match any declared foreign key relationship in the ontology. The query still runs, but OBQC suggests the correct join condition from the ontology.

```sql
-- WARNING: JOIN condition may not match declared FK relationship
-- Suggested: orders.customer_id = customers.id
SELECT * FROM customers JOIN orders ON customers.name = orders.note;
```

### Type compatibility (WARNING)

Checks `WHERE` and `ON` clause comparisons for type mismatches. Types are inferred from the ontology's XSD type mappings (`RDFS.range`) and categorized as numeric, string, temporal, or boolean.

```sql
-- WARNING: Type mismatch: string vs integer
SELECT * FROM products WHERE name > 100;
```

Comparisons within the same category are allowed (e.g. integer vs. decimal). Comparisons across categories produce a warning.

### Aggregation correctness (ERROR)

When aggregate functions (`SUM`, `COUNT`, `AVG`, `MIN`, `MAX`) are used, OBQC checks that all non-aggregated columns in `SELECT` appear in `GROUP BY`.

```sql
-- ERROR: Column 'name' in SELECT with aggregation but no GROUP BY
SELECT name, COUNT(*) FROM customers;

-- ERROR: Column 'status' not in GROUP BY clause
SELECT status, region, SUM(amount) FROM orders GROUP BY region;
```

### Fan-trap detection (WARNING)

Detects when a query aggregates across two or more one-to-many joins. This is the classic fan-trap pattern where rows multiply silently, producing inflated aggregation results.

```sql
-- WARNING: Potential fan-trap: 2 one-to-many joins with aggregation
SELECT c.name, SUM(o.amount), COUNT(r.id)
FROM customers c
JOIN orders o ON c.id = o.customer_id
JOIN reviews r ON c.id = r.customer_id
GROUP BY c.name;
```

OBQC suggests using `UNION ALL` to aggregate each fact table separately, or CTEs to pre-aggregate before joining. See [Fan-Trap Prevention](fan-trap-prevention.md) for detailed patterns.

## How the LLM uses OBQC

OBQC results are returned as structured data in the tool response, not displayed directly to the user. The LLM receives:

- `obqc_valid`: overall pass/fail
- `obqc_issues`: list of issues with type, severity, message, location, and suggestion
- `fan_trap_risk`: boolean flag
- `obqc_error_count` / `obqc_warning_count`: summary counts

When `execute_sql_query` blocks a query, the LLM sees the error details and suggestions, and can revise the SQL and retry -- often without the user ever seeing the failed attempt.

When warnings are returned alongside successful results, the LLM can decide whether to mention them to the user (e.g. "Note: this query joins across two one-to-many relationships, which may inflate totals").

## Requirements for OBQC

OBQC relies on `oba:` namespace annotations in the ontology. These are automatically included when using `generate_ontology` or `load_my_ontology` with an OrionBelt-generated ontology.

**Required annotations per OWL Class (table):**

| Annotation | Purpose |
|------------|---------|
| `oba:tableName` | Actual database table name |
| `oba:schemaName` | Database schema (defaults to "public") |

**Required annotations per OWL DatatypeProperty (column):**

| Annotation | Purpose |
|------------|---------|
| `oba:columnName` | Actual column name |
| `oba:tableName` | Parent table name |
| `oba:sqlDataType` | SQL type (e.g. `VARCHAR(100)`, `INTEGER`) |
| `oba:isPrimaryKey` | Boolean |
| `oba:isForeignKey` | Boolean |
| `RDFS.range` | XSD type for type compatibility checks |

**Required annotations per OWL ObjectProperty (relationship):**

| Annotation | Purpose |
|------------|---------|
| `oba:foreignKeyColumn` | FK column name |
| `oba:referencedTable` | Target table name |
| `oba:referencedColumn` | Target column name |
| `oba:relationshipType` | `many_to_one` or `one_to_many` |
| `oba:sqlJoinCondition` | SQL join expression |

If a loaded ontology lacks these annotations, OBQC reports `ontology_compatible: false` and falls back to basic syntax checking only.
